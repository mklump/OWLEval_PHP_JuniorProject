A Concise History of the Changes to Our Original Design

Original Design
Connected to OIT’s Database to cull course and instructor information.
Would be implemented completely in PHP
Would allow instructors to augment their own evaluations, and would require administrators to approve those augmentations.

Initial Modifications to our Requirements
Access to OIT’s database proved problematic
Bureaucratic Nightmare
OIT’s Database is Oracle; we don’t have access to Oracle for development purposes
OIT’s DB has 1200+ tables

Solution
We would not connect at all to OIT’s Database.
There is very little information that our application actually needs from OIT’s DB.
By eliminating our requirement to work with the existing database, we’d increase the amount of work that the administrator would have to do.

Further Changes
To increase Database performance, the interface to our DB would be implemented in Perl using FastCGI
FastCGI allows a script to remain in binary format in memory, and would maintain an open DB connection which all of our other objects could use.

Augmentation Approval Unnecessary
It was determined that the instructors could be trusted to write augmentations that would be appropriate, and that by having the administrators approve all augmentations created a lot of unnecessary work.
The requirement for administrator approval of augmentations was dropped

Other Languages
It was suggested that we consider using .Net for our project and C#, as more of us have familiarity with that language than with PHP or Perl.
The logistical considerations of using IIS-centric technology were brought up, and it was suggested that we consider using Java Servlets with Tomcat server as a more practical option, considering our available resources.

The Results
Considering that these suggestions were made so late in the term, after weighing the benefits of these changes versus the modifications we’d have to make to our design, we opted to stay the course with PHP and Perl.

PHP4 Obviates FastCGI
Researching the features of the PHP language, a library was discovered that maintains a persistent DB connection from one script on a server to the next.
This performs the task that FastCGI was brought in to the design to do, without the hellacious interface that FastCGI would’ve required.
FastCGI was dropped from the design

Wrap-up
We ended up with a design that was very similar to what we’d started out with, sans a few problematic requirements.
